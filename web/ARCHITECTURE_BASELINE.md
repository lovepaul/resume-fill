# 架构基线（deploy + web）

本文档记录当前系统的可维护性、安全性、稳定性基线，作为后续 Clean Code 改造与里程碑验收的统一参考。

## 1. 当前架构概览

### 前端
- 主站：`web/frontend/src/App.jsx`（React 单组件主流程）
- 子页面：`web/frontend/public/stats/*` 与 `web/frontend/public/nvwa/*`（原生 JS）
- 构建：Vite，`public` 目录静态拷贝到 `dist`

### 后端
- 单入口：`web/backend/app.py`（FastAPI + 路由 + 状态 + 鉴权 + 清理 + 子进程调用）
- 任务模型：进程内字典保存状态，线程执行转换/蒸馏
- 运行目录：`WEB_RUNTIME_DIR`（默认 Linux 为 `/var/lib/resume-web/runtime`）

### 部署与运维
- 脚本入口：`deploy/linux/install-from-github.sh` -> `deploy/linux/bootstrap.sh`
- 服务编排：systemd + nginx + cron
- 管理命令：`deploy/linux/manage.sh`

## 2. 主要问题基线

### 前端基线问题
- 重复实现：主站、stats、nvwa 存在重复的 token/API/错误处理逻辑（G5）
- 魔法数分散：轮询间隔、冷却时间、分页尺寸分散硬编码（G25）
- 入口职责偏重：`App.jsx` 同时承担 UI、网络、轮询、下载、历史存储（G30）

### 后端基线问题
- 单文件过大：`app.py` 耦合路由与业务细节，难以测试与演进（G30/G34）
- 会话与限流逻辑重复：stats/nvwa 分别实现近似会话逻辑（G5）
- 状态在进程内：多 worker 或重启恢复能力有限

### DevOps 基线问题
- 默认弱口令风险：部署参数与示例值存在弱默认配置
- 配置覆盖前缺少明确备份步骤
- 运行目录清理任务日志默认丢弃，故障可观测性弱

## 3. 目标状态（M1 对齐）

### 代码质量
- 前端建立共享工具层，消除重复逻辑与魔法数
- 后端抽离复用能力（会话、限流）到独立模块，降低 `app.py` 复杂度

### 安全基线
- 部署阶段对弱口令与关键变量做显式校验
- 环境变量文件权限保持最小化（600）

### 运维基线
- 清理任务具备可追踪日志
- 部署前具备配置快照备份

## 4. 验收观察指标
- 重复逻辑减少：前端跨入口重复工具函数收敛到共享模块
- 主入口瘦身：后端公共逻辑迁出 `app.py`
- 安全门禁：部署脚本在不安全配置下直接失败
- 可观测提升：定时任务失败可通过日志定位
